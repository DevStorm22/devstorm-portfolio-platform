# 1. Purpose

This document defines the complete API contract for the DevStorm Portfolio Platform.

It specifies the structure, standards, conventions, and behavior of every REST API endpoint that will be implemented during development.

This document serves as the authoritative reference for backend implementation, frontend integration, testing, and future API maintenance.

# 3. API Design Objectives

The API has been designed with the following objectives.

- Consistent endpoint structure.
- Predictable request and response formats.
- Secure administrative operations.
- Clear separation between public and protected endpoints.
- Easy frontend integration.
- High maintainability.
- Future API version support.
- Standards compliant REST architecture.

---

# 4. REST Architectural Principles

The API follows REST (Representational State Transfer) principles.

---

## Resource-Oriented Design

Each endpoint represents a business resource.

Examples:

```text
/projects

/skills

/certifications

/cyber-reports

/contact
```

---

## Stateless Communication

Every request contains all information required to process it.

No server-side session state is maintained.

---

## Uniform Interface

All resources follow consistent naming and behavior.

Example:

```text
GET

POST

PUT

PATCH

DELETE
```

---

## Client-Server Separation

The frontend communicates only through REST APIs.

Direct database access is prohibited.

---

## Layered Architecture

The API follows the application architecture.

```text
Client

↓

Route

↓

Middleware

↓

Controller

↓

Service

↓

Repository

↓

MongoDB
```

---

# 5. API Design Principles

The following principles apply to every endpoint.

---

## Consistency

Endpoints follow identical request and response patterns.

---

## Simplicity

URLs remain short, descriptive, and predictable.

---

## Security

Protected endpoints require authentication.

---

## Validation

Every request is validated before reaching business logic.

---

## Standardized Responses

Every endpoint returns a consistent JSON structure.

---

## Predictable Errors

Errors always follow the same response format.

---

# 6. API Versioning Strategy

The application adopts URI versioning.

---

## Base URL

```text
/api/v1
```

---

## Example Endpoints

```text
/api/v1/projects

/api/v1/auth/login

/api/v1/skills

/api/v1/contact
```

---

## Future Versioning

Future releases may introduce:

```text
/api/v2
```

without breaking existing clients.

---

## Versioning Principles

- Version included in URL.
- Breaking changes require a new version.
- Minor additions remain within the current version.
- Deprecated versions receive migration documentation.

---

# 7. URL Design Standards

## Resource Naming

Use plural nouns.

Correct:

```text
/projects

/skills

/certifications
```

Incorrect:

```text
/getProjects

/createProject

/projectList
```

---

## Lowercase URLs

Correct:

```text
/cyber-reports
```

Incorrect:

```text
/CyberReports
```

---

## Hyphenated Resources

Use hyphens instead of camelCase.

Correct:

```text
/social-links

/cyber-reports
```

Incorrect:

```text
/socialLinks

/cyberReports
```

---

## Nested Resources

Future nested resources:

```text
/projects/{id}/images
```

# 10. Authentication Strategy

## Overview

The DevStorm Portfolio Platform uses JSON Web Token (JWT) authentication to secure administrative APIs.

Public portfolio resources remain accessible without authentication, while administrative operations require a valid access token.

---

## Authentication Flow

```text
Administrator

↓

Login Request

↓

Authentication API

↓

Credential Verification

↓

JWT Generation

↓

Client Stores Token

↓

Authenticated API Requests

↓

JWT Verification

↓

Protected Resource
```

---

## Authentication Types

| Endpoint Type | Authentication |
|---------------|----------------|
| Public Portfolio | Not Required |
| Contact Form | Not Required |
| Admin Dashboard | Required |
| CRUD Operations | Required |
| Settings | Required |
| Resume Upload | Required |

---

## Authorization Roles

Current release supports:

| Role | Permissions |
|------|-------------|
| Visitor | Read-only public APIs |
| Administrator | Full CRUD access |

Future versions may introduce:

- Editor
- Moderator
- Super Administrator

# 11. Request Standards

Every API request follows a common structure.

---

## Standard Headers

```http
Content-Type: application/json
Accept: application/json
```

Authenticated requests additionally include:

```http
Authorization: Bearer <JWT_TOKEN>
```

---

## Path Parameters

Example:

```http
GET /api/v1/projects/{projectId}
```

---

## Query Parameters

Supported parameters:

```text
?page=1

&limit=10

&sort=-createdAt

&search=react

&category=frontend
```

---

## Request Body

Example:

```json
{
  "title": "DevStorm Portfolio",
  "category": "Web Development"
}
```

---

## Request Principles

- UTF-8 encoding.
- JSON payloads.
- ISO-8601 date format.
- Consistent field naming.
- Validate before processing.

---

# 12. Response Standards

Every endpoint returns a standardized response.

---

## Success Response

```json
{
  "success": true,
  "message": "Operation completed successfully.",
  "data": {},
  "meta": {}
}
```

---

## Collection Response

```json
{
  "success": true,
  "message": "Projects retrieved successfully.",
  "data": [],
  "meta": {
    "page": 1,
    "limit": 10,
    "total": 42
  }
}
```

---

## Error Response

```json
{
  "success": false,
  "message": "Validation failed.",
  "errors": [],
  "statusCode": 400
}
```

---

## Response Principles

- Predictable structure.
- Human-readable messages.
- Machine-readable data.
- Pagination metadata where applicable.
- Appropriate HTTP status codes.

---

# 13. HTTP Status Code Standards

The API uses standard HTTP status codes.

| Status Code | Meaning |
|-------------|----------|
| 200 | Success |
| 201 | Resource Created |
| 204 | No Content |
| 400 | Bad Request |
| 401 | Unauthorized |
| 403 | Forbidden |
| 404 | Resource Not Found |
| 409 | Conflict |
| 422 | Validation Error |
| 429 | Too Many Requests |
| 500 | Internal Server Error |

---

## Status Code Principles

- Always use the most appropriate status code.
- Avoid returning 200 for failed operations.
- Validation errors return 422 where applicable.
- Authentication failures return 401.
- Authorization failures return 403.

Validation occurs before business logic execution.

---

## Validation Layers

```text

↓

Controller

↓

Business Logic

↓

MongoDB
```

## Validation Responsibilities

- Email format

---

### Zod

- Data types
- Enums
- Business constraints

---

### Mongoose

- Required fields
- Unique constraints
- Default values

# 17. Correlation ID Strategy

Every request should be assigned a unique correlation ID.

Example header:

```http
X-Request-ID: 5f4fcb73-6dd7-4ef8-a3bc-d98c2e91f8c1
```

Benefits:

- Easier debugging
- Log correlation
- Request tracing
- Production diagnostics

---

# 18. Idempotency Guidelines

Safe operations should remain idempotent.

| Method | Idempotent |
|---------|------------|
| GET | Yes |
| PUT | Yes |
| PATCH | Usually |
| DELETE | Yes |
| POST | No |

Future payment or webhook integrations may introduce explicit idempotency keys.
