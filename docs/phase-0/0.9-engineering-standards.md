# 1. Purpose

This document establishes the engineering standards, technology choices, development practices, and quality guidelines for the DevStorm Portfolio Platform.

It serves as the primary engineering reference for all contributors throughout the software development lifecycle.

The objective is to ensure consistency, maintainability, scalability, and long-term sustainability of the project.

---

# 2. Scope

This document defines standards for:

- Technology stack
- Development workflow
- Coding conventions
- Git workflow
- Branch management
- Commit conventions
- Folder organization
- Testing
- Security
- Documentation
- Code reviews
- Release management

---

# 3. Engineering Objectives

The engineering standards are designed to achieve the following objectives.

- Maintain a clean and modular architecture.
- Improve code readability.
- Reduce technical debt.
- Encourage reusable components.
- Standardize development practices.
- Simplify onboarding.
- Improve maintainability.
- Support future scalability.

---

# 4. Engineering Principles

The project follows the following engineering principles.

---

## Simplicity

Solutions should remain simple unless additional complexity provides measurable value.

---

## Maintainability

Code should prioritize readability and long-term maintenance over premature optimization.

---

## Reusability

Reusable modules shall be preferred over duplicated implementations.

---

## Consistency

Naming, formatting, structure, and workflows must remain consistent across the project.

---

## Security First

Security considerations must be incorporated throughout development rather than added later.

---

## Documentation Driven

Major architectural and implementation decisions shall be documented before development.

---

## Incremental Development

Features shall be implemented in small, reviewable increments with meaningful Git history.

# 16. Naming Conventions

Consistency in naming improves readability and maintainability.

---

## Files

Use lowercase kebab-case.

Examples:

```text
project.controller.ts

user.service.ts

error-handler.ts
```

---

## Folders

Use lowercase kebab-case.

Examples:

```text
shared/

middleware/

project-management/
```

---

## React Components

Use PascalCase.

Examples:

```text
Navbar.tsx

ProjectCard.tsx

Footer.tsx
```

---

## TypeScript Interfaces

Prefix with `I`.

Examples:

```text
IProject

IUser

IApiResponse
```

---

## Types

Suffix with `Type`.

Examples:

```text
UserType

RoleType

ProjectType
```

---

## Constants

Use UPPER_SNAKE_CASE.

Examples:

```text
MAX_UPLOAD_SIZE

JWT_SECRET_KEY

DEFAULT_PAGE_SIZE
```

---

## Variables & Functions

Use camelCase.

Examples:

```text
getProjects()

createUser()

projectService
```

---

## Classes

Use PascalCase.

Examples:

```text
ProjectService

AuthController

DatabaseConnection
```

---

## API Routes

Use lowercase plural resources.

Examples:

```text
/api/v1/projects

/api/v1/skills

/api/v1/contact
```

# 17. Import Organization

Imports shall follow the order below.

```text
1. Node.js built-in modules

2. Third-party packages

3. Internal aliases

4. Relative imports

5. Styles
```

Example:

```typescript
import fs from "node:fs";

import express from "express";

import { ProjectService } from "@/modules/projects";

import { validateRequest } from "../middleware";

import "./styles.css";
```

# 18. Git Workflow

The project adopts a structured Git workflow to ensure traceability, maintainability, and collaboration readiness.

---

## Workflow Overview

```text
main

↓

Create Feature Branch

↓

Development

↓

Local Testing

↓

Commit Changes

↓

Push Branch

↓

Create Pull Request

↓

Code Review

↓

Approval

↓

Merge into main

↓

Delete Branch
```

---

## Workflow Principles

- Never commit directly to `main`.
- Every task shall be completed in its own branch.
- Every branch shall have a corresponding Pull Request.
- Pull Requests shall be reviewed before merging.
- Commits shall remain small and meaningful.
- Merge only after all quality checks pass.

---

# 20. Commit Message Convention

The project follows the Conventional Commits specification.

---

## Format

```text
<type>(<scope>): <description>
```

---

## Examples

```text
feat(auth): implement JWT authentication

feat(projects): add project CRUD endpoints

fix(validation): handle invalid ObjectId

docs(phase-0): complete module 0.9 engineering standards

refactor(project): simplify repository logic

test(auth): add login integration tests

style(ui): format React components

chore(deps): update development dependencies
```

---

## Allowed Types

| Type | Purpose |
|------|---------|
| feat | New feature |
| fix | Bug fix |
| docs | Documentation |
| style | Formatting |
| refactor | Code improvement |
| test | Testing |
| chore | Maintenance |
| perf | Performance improvement |
| ci | CI/CD changes |
| build | Build configuration |

---

# 21. Pull Request Workflow

Every branch shall be integrated through a Pull Request.

---

## PR Lifecycle

```text
Feature Branch

↓

Push to GitHub

↓

Open Pull Request

↓

Automated Checks

↓

Code Review

↓

Approval

↓

Merge

↓

Delete Branch
```

---

## Pull Request Requirements

A Pull Request must include:

- Clear title
- Summary of changes
- Related module or issue
- Testing status
- Documentation updates
- Reviewer checklist

---

## Merge Strategy

Preferred merge method:

```text
Squash and Merge
```

Reasons:

- Cleaner history
- One commit per completed task
- Easier rollback
- Better release notes

---

# 22. Code Review Standards

All Pull Requests should be reviewed against the following criteria.

---

## Functional Review

- Meets requirements
- No broken functionality
- Handles edge cases

---

## Code Quality Review

- Readable
- Modular
- No duplicated logic
- Proper naming
- Minimal complexity

---

## Security Review

- Input validation
- Authorization checks
- Secure secrets handling
- Proper error handling

---

## Documentation Review

- README updated if required
- API documentation updated
- Comments only where necessary
