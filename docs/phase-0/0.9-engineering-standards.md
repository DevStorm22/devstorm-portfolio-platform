# 1. Purpose

This document establishes the engineering standards, technology choices, development practices, and quality guidelines for the DevStorm Portfolio Platform.

It serves as the primary engineering reference for all contributors throughout the software development lifecycle.

The objective is to ensure consistency, maintainability, scalability, and long-term sustainability of the project.

---

# 2. Scope

This document defines standards for:

- Technology stack
- Development workflow
- Coding conventions
- Git workflow
- Branch management
- Commit conventions
- Folder organization
- Testing
- Security
- Documentation
- Code reviews
- Release management

---

# 3. Engineering Objectives

The engineering standards are designed to achieve the following objectives.

- Maintain a clean and modular architecture.
- Improve code readability.
- Reduce technical debt.
- Encourage reusable components.
- Standardize development practices.
- Simplify onboarding.
- Improve maintainability.
- Support future scalability.

---

# 4. Engineering Principles

The project follows the following engineering principles.

---

## Simplicity

Solutions should remain simple unless additional complexity provides measurable value.

---

## Maintainability

Code should prioritize readability and long-term maintenance over premature optimization.

---

## Reusability

Reusable modules shall be preferred over duplicated implementations.

---

## Consistency

Naming, formatting, structure, and workflows must remain consistent across the project.

---

## Security First

Security considerations must be incorporated throughout development rather than added later.

---

## Documentation Driven

Major architectural and implementation decisions shall be documented before development.

---

## Incremental Development

Features shall be implemented in small, reviewable increments with meaningful Git history.

# 18. Git Workflow

The project adopts a structured Git workflow to ensure traceability, maintainability, and collaboration readiness.

---

## Workflow Overview

```text
main

↓

Create Feature Branch

↓

Development

↓

Local Testing

↓

Commit Changes

↓

Push Branch

↓

Create Pull Request

↓

Code Review

↓

Approval

↓

Merge into main

↓

Delete Branch
```

---

## Workflow Principles

- Never commit directly to `main`.
- Every task shall be completed in its own branch.
- Every branch shall have a corresponding Pull Request.
- Pull Requests shall be reviewed before merging.
- Commits shall remain small and meaningful.
- Merge only after all quality checks pass.

# 29. General Coding Standards

All source code shall follow consistent engineering principles to ensure readability, maintainability, and scalability.

---

## Core Principles

The project adopts the following software engineering principles:

- SOLID
- DRY (Don't Repeat Yourself)
- KISS (Keep It Simple, Stupid)
- YAGNI (You Aren't Gonna Need It)
- Separation of Concerns
- Single Responsibility Principle
- Composition over Inheritance

---

## General Rules

- Prefer readable code over clever code.
- Keep functions small and focused.
- Avoid duplicated logic.
- Write self-explanatory code.
- Eliminate unused variables and imports.
- Use early returns to reduce nesting.
- Avoid hardcoded values.
- Store reusable values as constants.

---

# 30. TypeScript Standards

TypeScript is mandatory across the frontend and backend.

---

## Compiler Configuration

- Enable `strict` mode.
- Avoid `any`.
- Prefer explicit return types for exported functions.
- Enable `noUnusedLocals`.
- Enable `noUnusedParameters`.

---

## Type Usage

Preferred order:

1. Interface
2. Type Alias
3. Generic Types

---

## Naming Standards

Interfaces:

```text
IProject
IUser
IApiResponse
```

Types:

```text
UserType
ProjectType
RoleType
```

Enums:

```text
UserRole
ProjectStatus
```

---

## Functions

Good

```typescript
function calculateScore(score: number): number
```

Avoid

```typescript
function calculateScore(score: any)
```

# 42. Security Standards

Security shall be integrated into every development stage.

---

## Authentication

- JWT Authentication
- Secure password hashing
- Protected admin routes

---

## Authorization

- Verify user permissions
- Restrict administrative endpoints
- Apply least privilege principle

---

## Input Validation

Validate:

- Body
- Query Parameters
- Route Parameters
- Uploaded Files

---

## Secret Management

Never commit:

- API Keys
- JWT Secrets
- Database Credentials
- Cloud Storage Credentials

---

## Dependency Security

- Review package updates regularly.
- Remove vulnerable dependencies.
- Prefer trusted open-source libraries.
