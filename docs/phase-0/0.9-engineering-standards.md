# 1. Purpose

This document establishes the engineering standards, technology choices, development practices, and quality guidelines for the DevStorm Portfolio Platform.

It serves as the primary engineering reference for all contributors throughout the software development lifecycle.

The objective is to ensure consistency, maintainability, scalability, and long-term sustainability of the project.

---

# 2. Scope

This document defines standards for:

- Technology stack
- Development workflow
- Coding conventions
- Git workflow
- Branch management
- Commit conventions
- Folder organization
- Testing
- Security
- Documentation
- Code reviews
- Release management

---

# 3. Engineering Objectives

The engineering standards are designed to achieve the following objectives.

- Maintain a clean and modular architecture.
- Improve code readability.
- Reduce technical debt.
- Encourage reusable components.
- Standardize development practices.
- Simplify onboarding.
- Improve maintainability.
- Support future scalability.

---

# 4. Engineering Principles

The project follows the following engineering principles.

---

## Simplicity

Solutions should remain simple unless additional complexity provides measurable value.

---

## Maintainability

Code should prioritize readability and long-term maintenance over premature optimization.

---

## Reusability

Reusable modules shall be preferred over duplicated implementations.

---

## Consistency

Naming, formatting, structure, and workflows must remain consistent across the project.

---

## Security First

Security considerations must be incorporated throughout development rather than added later.

---

## Documentation Driven

Major architectural and implementation decisions shall be documented before development.

---

## Incremental Development

Features shall be implemented in small, reviewable increments with meaningful Git history.

# 6. Technology Selection Rationale

The following technologies were selected based on project requirements and long-term maintainability.

---

## React

Selected for its component-based architecture, ecosystem maturity, and strong TypeScript support.

---

## TypeScript

Provides static type checking, improved tooling, and reduced runtime errors.

---

## Express.js

Lightweight framework suitable for REST API development with excellent middleware support.

---

## MongoDB Atlas

Provides a managed, scalable NoSQL database service with strong integration into the MERN stack.

---

## Mongoose

Offers schema validation, middleware support, and a structured data access layer.

---

## Zod

Enables runtime validation with TypeScript-first schema definitions.

---

## JWT

Supports stateless authentication for secure API access.

---

## Tailwind CSS

Allows rapid UI development while maintaining a consistent design system.

# 7. Development Philosophy

Development shall follow the principles below.

- Documentation before implementation.
- Feature-first development.
- Small, focused commits.
- Peer-review-ready pull requests.
- Consistent coding standards.
- Test critical functionality.
- Refactor when justified.
- Avoid unnecessary dependencies.

# 15. Environment Variable Standards

Environment variables shall never be committed to source control.

---

## Backend

```text
.env

.env.example

.env.production
```

---

## Frontend

```text
.env

.env.example
```

---

## Naming Convention

Environment variables shall use uppercase snake_case.

Examples:

```text
PORT

DATABASE_URL

JWT_SECRET

JWT_EXPIRES_IN

CLOUDINARY_API_KEY

CLIENT_URL
```

---

# 16. Naming Conventions

Consistency in naming improves readability and maintainability.

---

## Files

Use lowercase kebab-case.

Examples:

```text
project.controller.ts

user.service.ts

error-handler.ts
```

---

## Folders

Use lowercase kebab-case.

Examples:

```text
shared/

middleware/

project-management/
```

---

## React Components

Use PascalCase.

Examples:

```text
Navbar.tsx

ProjectCard.tsx

Footer.tsx
```

---

## TypeScript Interfaces

Prefix with `I`.

Examples:

```text
IProject

IUser

IApiResponse
```

---

## Types

Suffix with `Type`.

Examples:

```text
UserType

RoleType

ProjectType
```

---

## Constants

Use UPPER_SNAKE_CASE.

Examples:

```text
MAX_UPLOAD_SIZE

JWT_SECRET_KEY

DEFAULT_PAGE_SIZE
```

---

## Variables & Functions

Use camelCase.

Examples:

```text
getProjects()

createUser()

projectService
```

---

## Classes

Use PascalCase.

Examples:

```text
ProjectService

AuthController

DatabaseConnection
```

---

## API Routes

Use lowercase plural resources.

Examples:

```text
/api/v1/projects

/api/v1/skills

/api/v1/contact
```

---

# 17. Import Organization

Imports shall follow the order below.

```text
1. Node.js built-in modules

2. Third-party packages

3. Internal aliases

4. Relative imports

5. Styles
```

Example:

```typescript
import fs from "node:fs";

import express from "express";

import { ProjectService } from "@/modules/projects";

import { validateRequest } from "../middleware";

import "./styles.css";
```
